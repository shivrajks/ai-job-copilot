# Product Decisions

**Purpose:** Records significant product and architecture decisions and their rationale.

---

## AD-001: Monorepo Structure

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use a monorepo with backend/, frontend/, docker/, and docs/ directories.

**Rationale:**
- Single repository simplifies development and CI/CD
- Shared documentation is always in sync with code
- Docker config co-located with service code
- No cross-repo version mismatch issues

---

## AD-002: UUID Primary Keys

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** All database primary keys use UUID v4 generated by PostgreSQL (`gen_random_uuid()`).

**Rationale:**
- No sequential ID enumeration vulnerability
- Safe for distributed systems and future sharding
- Client can generate IDs without DB round-trip (though we use DB generation)
- Consistent across all entities

---

## AD-003: JWT with Refresh Token Rotation

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use short-lived access tokens (15 min) with longer-lived refresh tokens (7 days). Implement refresh token rotation: each refresh revokes the previous token.

**Rationale:**
- Short access tokens limit damage if stolen
- Rotation detects token theft (reuse of revoked token triggers full session revocation)
- No server-side session storage needed
- Stateless authentication aligned with REST principles

---

## AD-004: BCrypt Password Hashing (Strength 12)

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Hash passwords with BCrypt at strength/work factor 12.

**Rationale:**
- Industry standard for password hashing
- Strength 12 provides ~250ms hash time on modern hardware (good UX/security balance)
- Built-in Spring Security support
- No dependency on additional libraries

---

## AD-005: DTOs as Inner Static Classes

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Group all DTOs for a domain into a single file as inner static classes (e.g., `AuthDtos.java` contains `RegisterRequest`, `LoginRequest`, `AuthResponse`, etc.).

**Rationale:**
- Reduces file count while maintaining clear organization
- DTOs are never used independently of their domain
- Easier to find all request/response shapes for a given controller
- Consistent with shadcn/ui pattern of co-location

---

## AD-006: No Field Injection — Constructor Injection Only

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use `@RequiredArgsConstructor` (Lombok) for constructor injection. Never use `@Autowired` on fields.

**Rationale:**
- Enables immutable dependencies (final fields)
- Explicit in tests (no reflection hacks)
- Clear dependency graph visible in class constructor
- Lombok reduces boilerplate

---

## AD-007: Glassmorphism Design System

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use glassmorphism design language with frosted glass surfaces, gradient accents (indigo→purple), Inter font, and subtle micro-interactions.

**Rationale:**
- Modern, premium SaaS aesthetic (inspired by Linear, Stripe, Vercel)
- Differentiates from generic Bootstrap/admin templates
- Works well in both light and dark mode
- CSS backdrop-filter is widely supported in modern browsers

---

## AD-008: AI Service Abstraction Layer

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Abstract all AI operations behind interfaces (AiService, TailoringAiService, CoverLetterAiService, InterviewAiService, JobDescriptionAiService) with Gemini and Mock implementations.

**Rationale:**
- Enables testing without API keys (Mock services return deterministic data)
- Easy to swap AI providers (future: OpenAI, Anthropic, local models)
- Clean separation between business logic and AI implementation
- Mock services allow full development without internet/Gemini access

---

## AD-009: Mock AI Services as Default

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Default `ai.provider=mock` in application.yml. Switch to `gemini` only when GEMINI_API_KEY is configured.

**Rationale:**
- Application runs out-of-the-box without any API keys
- Prevents accidental API usage/costs during development
- Mock services produce realistic but deterministic responses
- Easy CI/CD testing without external dependencies

---

## AD-010: Flyway for Database Migrations

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use Flyway for all database schema changes with `ddl-auto: validate` (never auto-create).

**Rationale:**
- Version-controlled, immutable migrations
- Rollback-safe with additive-only changes (ALTER TABLE ADD COLUMN IF NOT EXISTS)
- Explicit schema evolution history
- Prevents drift between local/dev/prod databases
- `validate` ensures entities match migrations at startup

---

## AD-011: Resume Storage — Local Filesystem

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Store uploaded resume files on local filesystem (`./uploads/resumes/`) with UUID-based filenames. FileStorageService interface allows future S3 swap.

**Rationale:**
- No AWS dependency for local development
- UUID filenames prevent path traversal and enumeration
- Interface abstraction enables S3 integration without code changes
- 5MB file size limit prevents abuse

---

## AD-012: JPA ddl-auto=validate with Flyway

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Set `spring.jpa.hibernate.ddl-auto=validate`. All schema changes go through Flyway migrations.

**Rationale:**
- Prevents Hibernate from silently creating/dropping columns
- Ensures entity annotations match actual database schema
- Catches entity-migration mismatches at startup
- Flyway is the single source of truth for schema

---

## AD-013: No Email Enumeration on Auth Endpoints

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Forgot-password and login endpoints return generic success messages regardless of whether the email exists.

**Rationale:**
- Prevents user enumeration attacks
- Standard security best practice
- Slightly degraded UX is acceptable for security

---

## AD-014: Single DTO File Per Domain

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Each domain has exactly one DTO file with all related request/response types as inner static classes (exception: AiDtos is shared across AI subdomain).

**Rationale:**
- Navigable: open one file to see all API shapes for a domain
- Consistent pattern across the backend
- Avoids dozens of tiny DTO files
- Inner classes are private to the domain DTO file

---

## AD-015: Zustand for Global State, React Query for Server State

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use Zustand for global client state (auth, UI preferences). Use @tanstack/react-query (via Zustand store wrappers) for server data fetching.

**Rationale:**
- Zustand is minimal (~1KB) and TypeScript-native
- React Query provides caching, refetching, loading/error states
- Separation of concerns: client state vs server cache
- Both are widely adopted in the React ecosystem

---

## AD-016: RESTful Plural Resource Names

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** All API endpoints use plural resource names (`/resumes`, `/applications`, `/jobs`). Sub-resources use nested paths (`/resumes/{id}/active`).

**Rationale:**
- Standard REST convention
- Consistent and predictable URL structure
- Self-documenting API design

---

## AD-017: Ownership Verification on All Resources

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Every service method that accesses user data verifies that the authenticated user owns the resource before reading or mutating.

**Rationale:**
- Prevents horizontal privilege escalation
- Defence in depth alongside JWT authentication
- User A cannot access User B's resumes, applications, or interviews

---

## AD-018: Two-Tier AI Model Strategy (Flash + Pro)

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use Gemini Flash for latency-sensitive operations (resume optimization, cover letter generation) and Gemini Pro for deep reasoning tasks (skill gap analysis, career suggestions, interview evaluation).

**Rationale:**
- Flash: faster, cheaper, sufficient for generation tasks
- Pro: better reasoning, necessary for evaluation and analysis
- Cost optimization: Flash for high-volume, Pro for quality-critical
- Configurable per service in AiConfig

---

## AD-019: No Server-Side Sessions

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Stateless authentication only. No HttpSession, no session affinity, no server-side session storage.

**Rationale:**
- Horizontally scalable (any instance can handle any request)
- JWT contains all necessary user context
- Session creation is disabled in SecurityConfig
- Redis is used for rate limiting, not sessions

---

## AD-020: Rate Limiting via Redis (Sliding Window)

**Date:** 2026-05-20
**Status:** Confirmed (backlog)

**Decision:** Implement per-user rate limiting using Redis sliding window. Free tier: 5 AI requests/month. Pro tier: 10,000/month.

**Rationale:**
- Prevents API abuse and cost overruns
- Free tier limits encourage upgrades
- Redis is already in the infrastructure stack
- Sliding window is more user-friendly than fixed window
- Rate limit headers in responses for transparency

**Note:** Config is defined in application.yml. Implementation is scoped as infrastructure backlog.

## AD-021: Separate Job and Application Entities

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Maintain two separate entities — `Job` (saved job listing with full metadata) and `Application` (tracking record with stage pipeline). Applications reference both a Job and a Resume.

**Rationale:**
- Separation of concerns: job listing data vs application tracking state
- Users can save jobs without applying
- Multiple applications can reference the same job
- Full job metadata (salary, employment type, work mode) stored independently

---

## AD-022: JSONB for Structured Analytical Data

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Store parsed resume data, extracted skills, and structured analysis as JSONB columns.

**Rationale:**
- Schema-less storage for variable AI output
- PostgreSQL JSONB supports indexing and querying
- No need for additional tables for every analysis type
- Easy to evolve as AI output formats change
- JdbcTypeCode(SqlTypes.JSON) maps cleanly between Java String and JSONB

---

## AD-023: Application and Job Entities Coexist

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Both `Application` (V1) and `Job` (V8) entities coexist. Applications track the stage pipeline for tracked opportunities. Jobs store saved listings with full metadata. The tracker frontend uses both, with Application as the primary tracker entity and Job for saved listings.

**Rationale:**
- Applications were created first for the stage-tracking pipeline
- Jobs were added later to support saved job listings with rich metadata
- They serve different but complementary use cases
- Migration V8 adds jobs without modifying applications

---

## AD-024: No Hard Deletes for User Data

**Date:** 2026-05-20
**Status:** Confirmed

**Decision:** Use soft deletes (`is_active` flag on resumes, cover letters) rather than hard deletes for user-facing data.

**Rationale:**
- Users can recover accidentally deleted items (future enhancement)
- Audit trail preserved for analytics
- CASCADE delete only for token/auth data where GDPR compliance is clear

---

## AD-025: Rate Limiting Configuration

**Date:** 2026-05-20
**Status:** Planned (infrastructure backlog)

**Decision:** Per-user AI rate limiting:
- Free tier: 5 requests per month
- Pro tier: 10,000 requests per month
- Implementation: Redis sliding window
- Rate limit headers in API responses

**Rationale:**
- Monetization mechanism for SaaS model
- Prevents abuse without hard blocks
- Configurable per feature type
- Transparent to users via headers
