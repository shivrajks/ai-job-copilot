# Architecture Rules

**Purpose:** Defines architectural principles, patterns, and constraints for all system components.

---

## 1. Backend Layering (Spring Boot)

Strict three-layer architecture:

```
Controller → Service → Repository
```

- Controllers only handle HTTP concerns (request mapping, response shaping, @Valid).
- Services contain business logic, are @Transactional where appropriate.
- Repositories extend Spring Data JPA interfaces; no custom SQL unless required for performance.
- Cross-cutting concerns (security, logging, exception handling) stay in config/ or filter/ packages.

## 2. Frontend Layering (Next.js App Router)

- Pages live in `src/app/` using the App Router file convention.
- Shared UI components live in `src/components/ui/`.
- Business-logic components co-locate with their page or live in `src/components/` by domain.
- State management uses Zustand for global auth state; React Query for server state.
- Utility functions go in `src/lib/`.

## 3. API Design

- All endpoints are prefixed with `/api/` (configured via `server.servlet.context-path`).
- Resource names are plural (`/resumes`, `/applications`).
- Standard HTTP methods: GET (read), POST (create), PUT (update/replace), DELETE (remove).
- Error responses use a consistent `ErrorResponse` shape with status, error, message, timestamp, path.
- Authenticated endpoints use `@AuthenticationPrincipal UserDetails` to extract the current user.

## 4. Data Layer

- Primary keys are UUIDs generated by the database (`gen_random_uuid()`).
- Audit timestamps (`created_at`, `updated_at`) are mandatory on every entity.
- Soft deletes preferred over hard deletes for user-facing data.
- JSONB columns store structured analytical data (parsed resumes, extracted skills).
- Flyway migrations are immutable once applied. Never edit an existing migration.

## 5. Security Boundaries

- Authentication is stateless (JWT). No HTTP sessions.
- Every user-facing resource must verify ownership before read or mutation.
- File storage paths never include the original filename. Use UUID-based filenames.
- Email addresses are normalized (lowercased, trimmed) before storage or comparison.
